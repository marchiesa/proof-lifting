# Problem 5: Euclidean GCD on bitvectors

## Result: VERIFIED (attempt 2 of 10)

## Problem
Compute GCD of two bv8 values using repeated subtraction. The spec uses a recursive ghost function Gcd on natural numbers, and the postcondition bridges bv8 and nat via `as int`.

## Why Z3 struggles
The recursive Gcd function on nat gets unfolded excessively by Z3, causing resource exhaustion (timeout). The theory bridging between bv8 modular subtraction and nat subtraction also creates difficulties.

## Solution approach

### Attempt 1 (FAILED - timeout)
Added the loop invariant `Gcd(a as int, b as int) == Gcd(a0 as int, b0 as int)` with helper lemmas `GcdStepA` and `GcdStepB` for the two branches. However, with Gcd transparent, Z3 attempted to unroll the recursive definition excessively and timed out after 30 seconds.

### Attempt 2 (SUCCESS)
Made Gcd `opaque` so Z3 cannot automatically unroll it. Then provided four focused lemmas that each `reveal Gcd()` in a controlled context:
- `GcdZeroLeft(b)`: proves `Gcd(0, b) == b`
- `GcdZeroRight(a)`: proves `Gcd(a, 0) == a`
- `GcdStepA(a, b)`: proves `Gcd(a, b) == Gcd(a-b, b)` when a >= b > 0
- `GcdStepB(a, b)`: proves `Gcd(a, b) == Gcd(a, b-a)` when b > a > 0

Each lemma body simply calls `reveal Gcd()`, which unfolds exactly one step. The method calls the appropriate lemma at each decision point: the early returns, each loop branch, and the final result extraction. This prevents Z3 from uncontrolled unfolding while providing exactly the facts needed.
