## Why It's Hard

The fundamental difficulty is the **optimality proof**. The spec requires proving that NO valid combination exists with fewer coins than the DP result. This requires:

1. **Restricted combination induction**: Define combinations using subsets of coins and prove the DP correctly computes the optimum for each subset.

2. **DP transition correctness**: When adding coin type `i`, the update `dp[j] = min(dp[j], dp[j - coins[i]] + 1)` is correct for unbounded usage. The self-referential nature (`dp[j]` depends on `dp[j - coins[i]]` in the SAME partially-updated table) makes this an induction within an induction.

3. **DotProduct/TotalCoins decomposition**: Need lemmas that decompose these functions when one coordinate changes, relating "use coin i one more time" to the mathematical structure.

4. **Universal quantification over ALL combinations**: The spec quantifies over arbitrary `seq<int>` combinations. Proving the lower bound requires showing that ANY valid combination can be decomposed into one using the DP's structure.

This would require approximately 100-200 additional lines of lemmas. The key missing lemma is the DP transition: "if dp[j-c] is optimal for amount j-c using coins[0..i+1], then dp[j-c]+1 is optimal for amount j using coins[0..i+1] if it's better than the current dp[j]."
